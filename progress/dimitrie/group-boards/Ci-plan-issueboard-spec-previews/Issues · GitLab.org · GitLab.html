    8.12 CI Plan

    The CI bugs that needs to be solved Runner bugs that needs to be solved

    The key features (includes a Frontend and Backend ) (sorted by severity):

    • Add support for Review Apps (requires a new issue describing a proposal): #20255 (comment 13669830) (requires Backend to receive an URL of app, requires Frontend to visualise it),
    • Use per-build token to act as an user triggering build: #18994 (#18993 , #18107 ): !5735 (requires review of concept and Backend to prepare changes),
    • Add an API to list manual actions on Environments and Deployments (required for Cog integration): !6094 (requires only Backend),
    • Improve CI e-mail notifications: #3976 (comment 13483955) !6019 (requires Backend to prepare service and Frontend to prepare e-mail templates),
    • Show deployed ago on Merge Requests when visualising Environments: #21652 (required Backend to prepare changes, and low amount of Frontend work)
    • Slack app: #20799

    The planned UX/Frontend improvements (@annabeldunstone @dimitrieh ):

    • Collapse similar builds on Pipeline Graph: #21286 (requires Backend to return a list of grouped builds, requires Frontend to implement collapsing and pop-over),
    • Client-Side Terminal Emulation with Line Numbers: #13205 : !5744 (requires Frontend to review and work with contributor to merge this)
    • Pipelines for commits: #18937 (requires Frontend)
    • Curve the lines in the pipeline graph: #21126 !6033
    • Cosmetic changes pipeline graph: #21604
    • Add "Triggered By" name and gravatar to pipelines: #19991

    The planned Backend improvements (@ayufan @godfat @grzesiek ):

    • Show all pipelines for upcoming Merge Request Versions (requires small amount of Backend)
    • Improve Cancel of Pipeline with external jobs: #17845
    • Improve pipeline duration: #18260 !6084 (@godfat )
    • Refactor Slack service to use pipeline_hooks : !5525
    • Update artifacts_size for old builds (!5076 ) (needs more discussion)
    • Improve pipeline fixtures in development environment !5838 (@grzesiek )

    The planned GitLab Runner improvements (@ayufan @tmaczukin @nick.thomas ):

    • Improve CI tests to include integration tests for different architectures: gitlab-org/gitlab-ci-multi-runner!276
    • Prepare monitoring recommendations with description for our infrastructure team
    • Move from fsouza/go-dockerclient to docker/engine-api gitlab-org/gitlab-ci-multi-runner!282
    • Fix artifact downloading with Windows cmd.exe shell: gitlab-org/gitlab-ci-multi-runner!284
    • Autodetect /ci in registration URL: gitlab-org/gitlab-ci-multi-runner!289

    Left-overs:

    • Add download buttons for downloading artifacts for a branch (@godfat ): #4255 , #14419 , !5142 (ready to be merged)
    • Support running builds on Kubernetes: gitlab-org/gitlab-ci-multi-runner!277 (ready to be merged)

      @ayufan I'm not opposed to #18178 , but I'm not sure it's as urgent/important as other changes. Is it solving particularly important customer needs? Also, the goal of #18178 seems to be about understanding queue delays holding back running a pipeline, rather than investigating runner queues. I want to make sure we try to solve the underlying problem. i.e. redesigning the runner views might be part of the solution, but isn't a goal itself. e.g. when I look at a merge request's pipelines, and I see it's pending, I'd like to see how long it has been queued for, and some information to help me understand how long it might be queued for such as the number of jobs to be processed and the number of runners processing them.

      For that matter, when I look at a finished pipeline, I should be able to see how long it was queued for, and even graph a history of queue times to help in capacity planning of runners.
